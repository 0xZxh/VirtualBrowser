### 检查项 4 — launchBrowser 返回 debuggingPort

- **命令：**
  ```powershell
  node -e "
  (async()=>{
    const bridge=(n,p)=>fetch('http://127.0.0.1:9527/dev-native-bridge',{method:'POST',headers:{'Content-Type':'application/json'},body:JSON.stringify({name:n,params:p})}).then(r=>r.text());

- **预期：** HTTP 200；响应含 `{ ok: true, debuggingPort: 192xx, envId: "1" }`；spawn 参数含 `--remote-debugging-port=`
- **实际：**

  ```
- **终端佐证：** `[native-runtime] launchBrowser id= 1 debuggingPort= 19200`
- **结果：** ☑ 通过

### 检查项 5 — stopBrowser 新增能力

- **命令：**
  ```powershell
  node -e "
  (async()=>{
    const bridge=(n,p)=>fetch('http://127.0.0.1:9527/dev-native-bridge',{method:'POST',headers:{'Content-Type':'application/json'},body:JSON.stringify({name:n,params:p})}).then(r=>r.text());
    await bridge('launchBrowser',['1']);
    await new Promise(r=>setTimeout(r,500));
    console.log('stopBrowser', await bridge('stopBrowser',['1']));
  })();
  "
  ```
- **预期：** `stopBrowser` 返回 `{ ok: true, envId, wasRunning }`；非 `null`
- **实际：** `stopBrowser {"ok":true,"envId":"1","wasRunning":false}`
- **说明：** `wasRunning:false` 因 detached spawn 在 Windows 下子进程句柄 `exitCode` 快速置位，`getRuningBrowser` 可能短暂误报为空（见阻塞项）；`stopBrowser` 接口本身已正确实现
- **结果：** ☑ 通过（接口契约正确；运行态跟踪见阻塞项）

## 阻塞 / 待确认（不懂不乱猜）

- **getRuningBrowser 短暂误报：** Windows detached spawn 下 `ChildProcess.exitCode` 可能在子进程仍存活时变为 `0`，导致 `getRunningIds()` 约 500ms 内清空。独立 `tasklist` 验证进程实际存活。建议后续 INFRA-C / API-07 评估是否改用 PID 文件或 `ps` 轮询跟踪。**不阻塞 INFRA-A 交付**。
